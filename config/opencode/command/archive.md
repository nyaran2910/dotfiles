---
description: 作業ディレクトリを整理し、private GitHub リポジトリとしてアーカイブする
agent: build
---

対象を GitHub の private リポジトリとしてアーカイブしてください。

ユーザーからの引数: `$ARGUMENTS`

引数に対象パスがあればそのディレクトリを使い、なければ現在の作業ディレクトリを使ってください。引数の残りは、授業名、課題名、年度などを判断するための補足情報として扱ってください。

以下を最後まで実行してください。

1. 対象のファイル構成、主要ファイル、既存ドキュメントを調べ、何の記録かを把握する。巨大な依存ディレクトリ、ビルド成果物、バイナリの中身は不要に読み込まない。
2. 対象が既存 Git worktree 内なら、そのトップレベルを対象として既存の履歴、ブランチ、リポジトリ名、remote をそのまま使う。別のリポジトリを作ったり、remote URL や公開範囲を勝手に変更したりしない。
3. Git リポジトリでなければ対象ディレクトリで Git を初期化する。
4. commit 前に、認証情報、秘密鍵、トークン、個人情報、`.env`、巨大ファイル、依存ディレクトリ、ビルド成果物など、公開・保存すべきでないファイルが含まれないか確認する。必要なら内容に合った `.gitignore` を最小限追加する。秘密情報らしきものが見つかった場合は commit せず、ユーザーに確認する。
5. README が存在しない場合だけ、全体を把握した上で `README.md` を作る。タイトル、目的または授業・課題の概要、内容・ディレクトリ構成、実行方法や閲覧方法が分かる場合はそれらを簡潔に記載する。分からない情報を創作しない。既存 README は上書きしない。
6. GitHub remote が存在しない場合、内容からリポジトリ名を決める。
   - 授業関連なら `<元号年>-<授業名>-<識別子>` とする。例: `r8-algorithms-final-report`。
   - 元号年は対象資料の年度を優先し、令和なら `r8` のように元号の英字頭文字を小文字にして年を続ける。資料とパスから年度を特定できなければ現在日付を確認する。
   - 授業名と識別子は内容を表す短い lowercase kebab-case にする。課題番号、テーマ、レポート種別など、一覧で区別できる情報を識別子に使う。
   - 授業関連でなければ、この形式を無理に使わず、内容を識別できる簡潔な lowercase kebab-case にする。
   - 候補が複数あり、意味のある名前を決める根拠が不足している場合だけ、候補を示してユーザーに短く確認する。
7. `gh auth status` を確認し、remote がない場合は `gh repo create <name> --private --source=. --remote=origin` で必ず private リポジトリを作る。同名リポジトリが既にある、認証できない、または remote に不整合がある場合は、別名や上書きを独断で行わず状況を報告する。
8. `git status` と差分を確認し、アーカイブ対象を追加して、内容に合う簡潔なメッセージで commit する。変更がなければ空 commit は作らない。他人の履歴を書き換えず、force push もしない。
9. upstream がなければ設定して通常の push を行う。既存 remote への push が拒否された場合、force push や remote の付け替えはせず状況を報告する。
10. 最後に、対象パス、使用または作成したリポジトリ名、GitHub URL、README・`.gitignore` の変更、commit、push の結果を簡潔にまとめる。

削除、履歴改変、既存 remote の置換、公開リポジトリへの可視性変更は行わないでください。
