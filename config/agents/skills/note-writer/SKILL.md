話題が広すぎて1記事に収まらない可能性がある場合、すぐに書き始めず、以下を確認する。

### 判断基準
- 話題が抽象的・広域（例：「AI全般」「マーケティング全般」）→ **確認必須**
- 話題が具体的・絞られている（例：「ChatGPTのプロンプト設計」）→ そのまま進んでOK

### 確認する内容（1記事 or 複数パート）

```
「[話題]」はかなり広いテーマですね。
以下のどちらで進めますか？

A) 1記事に絞る（特定の切り口に絞ってコンパクトにまとめる）
B) 複数パートに分ける（連載形式で各回テーマを分ける）

Bの場合、全体のシリーズ構成案も一緒に提示します。
```

## ステップ3：構成を提案する（本文執筆前に必ずレビューをもらう）

ユーザーからGoサインが出たら、**本文を書く前に**見出し構成を提示してレビューをもらう。

### 構成フォーマット

```
## [記事タイトル案]

【入り】
読者への問いかけ＋記事の目的を簡単に説明（2〜3文のイメージ）

【目次】
1. [レベル2見出し]
   1-1. [レベル3見出し（あれば）]
   1-2. [レベル3見出し（あれば）]
2. [レベル2見出し]
   ...
N. おわりに

※ 伝えたいメッセージの対応：
- メッセージ①「〜」→ [対応する見出し]
- メッセージ②「〜」→ [対応する見出し]
...

この構成でよければ「OK」とお伝えください。修正があれば教えてください。
```

---

## ステップ4：本文を執筆する

ユーザーから「OK」が出たら本文を執筆する。

### フォーマットルール（厳守）

#### 入り文章
- 冒頭に読者への問いかけや共感フレーズ（例：「〜って気になりませんか？」）
- 記事の目的・読むと得られることを一言で説明
- その後に **目次**（見出し一覧）を箇条書きで記載

#### 本文構造
- 入り文章以外は **すべてレベル2見出し（##）の中に入れる**
- レベル2見出しの外に本文を書かない

#### 見出しのルール

**レベル2見出し（## ）のあとには必ず区切り線を入れる**

```markdown
## 見出しタイトル

---

本文...
```

**レベル3見出し（### ）は内容が複数に分かれる場合に使う**

```markdown
## 大きなテーマ

---

### 具体的なポイント①

内容...

### 具体的なポイント②

内容...
```

#### おわりに（必須）
- 最後のレベル2見出しは必ず `## おわりに`
- 構成例：
  1. 記事のまとめ・振り返り（1〜2文）
  2. 読者への感謝（「読んでくれてありがとうございました！」）
  3. 関連記事・次回予告への誘導（「こんな記事もあります」「次回は〜について書きます」）

---

## 記事フォーマット全体イメージ

```markdown
[入り文章：問いかけ＋目的説明]

**目次**
- 見出し1
- 見出し2
- ...
- おわりに

## 見出し1

---

本文...

### 小見出し1-1

内容...

### 小見出し1-2

内容...

## 見出し2

---

本文...

## おわりに

---

まとめ＋感謝＋関連記事への誘導
```

---

## 注意事項

- 構成レビュー前に本文を書き始めない
- レベル2見出しの直後には必ず `---`（区切り線）を入れる
- 入り文章と目次以外は、すべて見出しの中に収める
- 記事の最後は必ず `## おわりに` で締める

## 目次について

noteはマークダウンの目次記法（`[[toc]]`等）に対応していないため、**本文中に目次を記載しない**。目次はユーザーが手動でnoteの機能から挿入する。構成提案時に目次を見出し一覧として提示するのはOKだが、本文執筆時には目次セクションを含めないこと。

## コードブロックと引用ブロックの使い分け

- **コードブロック（` ``` `）**：実際のコード（プログラム、コマンド等）を示す場合のみ使用
- **引用ブロック（`>`）**：例文・会話例・概念的なフロー・プロンプト例など「例示」全般に使用

```markdown
// ❌ 例示にコードブロックは使わない
❌ 「さっき言ってた件でアイデアちょうだい」

// ✅ 例示には引用ブロックを使う
> ❌ 「さっき言ってた件でアイデアちょうだい」
> ✅ 「具体的な前提情報を含むプロンプト」
```
